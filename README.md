# Skarkix: Ridges Competitive Coding Agent

Skarkix is a highly optimised, autonomous coding agent designed specifically for the **Ridges Competition 23**. It is built to seamlessly navigate complex, multi-language codebases (SWE-bench, Polyglot, InfiniteSWE) and deliver robust, verified patches entirely from within isolated container sandboxes.

This repository contains the single required submission file: `agent.py`.

---

## Architecture Overview

`agent.py` is entirely self-contained, running without external libraries (aside from `requests` and Python standard library) to comply strictly with Ridges sandbox constraints. 

### 1. The Entry Point: `agent_main()`
The platform invokes the agent via the `agent_main(input_dict: dict) -> dict` function. 
1. It extracts the `problem_statement` and generates a persistent `run_id`.
2. It initialises `AgentConfig` to set up operational constraints (wall-clock timeouts, maximum step limits, target model selection).
3. It creates an instance of `SkarkixAgent` and calls its `run()` loop.
4. It finally returns the exact format required by the backend: `{"patch": "<git diff string>"}`.

### 2. Network & Sandbox Proxy (`inference()`)
Because Ridges runs agents inside locked-down sandboxes, raw API calls to OpenRouter or external LLM providers are blocked.
* Skarkix bypasses this limitation by routing traffic through the local host's inference gateway using the `$SANDBOX_PROXY_URL` environment variable (defaulting to `http://localhost:1234/agents/inference`).
* The proxy securely signs and forwards requests on the agent's behalf.
* It uses advanced prompt-caching headers (`_apply_prompt_cache_markers`) to drastically reduce the cost and latency of deep, multi-turn reasoning loops.

### 3. Execution Environment (`WorkspaceEnvironment`)
Skarkix does not use simple one-off subprocess commands. Instead, it spins up a continuous `pexpect` shell session.
* **Persistent State:** Environment variables, working directories (`cd`), and aliases are preserved between steps.
* **Timeout & Recovery:** If a command hangs (e.g., a rogue infinite loop), the shell terminates the hanging process and cleanly recovers.

## Action Parsing & Editing Tools

The core intelligence of Skarkix lies in how it translates LLM text generation into concrete codebase mutations. The agent parses specific XML-like action blocks generated by the LLM.

### The Standard Shell: `bash`
The LLM can output ````bash ... ```` blocks. Skarkix executes these precisely as provided and feeds stdout/stderr back into the conversation state.

### Single Edit: `skarkix_edit`
For simple edits, the LLM emits a search-and-replace block. It looks for the exact `<<<<` target content and replaces it with the `====` replacement content.

### Multi-Edit: `skarkix_multi_edit`
This is Skarkix's secret weapon. It allows the agent to safely perform complex, non-contiguous edits across massive files in a single turn. 
* It accepts multiple chunk replacements per file.
* It parses the AST (Abstract Syntax Tree) implicitly to ensure edits don't break surrounding indentation or scoping.
* It aggressively validates that the target string (`TargetContent`) exactly matches the target file to prevent hallucinated overwrites.

## Validation & Guardrails

Skarkix is paranoid about producing broken patches. It employs multiple tiers of verification before yielding a final answer.

1. **Format Error Escalation (`_format_error_escalation`)**
   If the LLM generates malformed tags or hallucinates commands, Skarkix issues a "strike". Strikes escalate in severity, forcing the LLM to read syntax documentation directly to correct its behaviour before retrying.
   
2. **Automated Pytest Triggers (`_run_module_pytest_on_submit`)**
   If the agent modifies a file and a corresponding test file exists (or it modifies the test file itself), Skarkix intercepts the action and automatically forces the agent to review a `pytest` run before submitting the final patch.
   
3. **Strict Patch Generation (`authoritative_worktree_patch`)**
   Instead of trusting the LLM to write diffs, Skarkix modifies the files directly on disk. At the very end of the run, it executes a raw `git diff` against the baseline to produce a perfectly formed, syntactically flawless patch artifact.
   
4. **Diff Linting & Sanity Checking (`_lint_submission_diff`)**
   Before returning, the patch is linted to ensure it doesn't just consist of accidental whitespace or trailing garbage.

## Evaluation Constraints & Time Limits

Skarkix respects the dynamic timeouts enforced by the Ridges evaluator:
- It tracks the `_effective_agent_wall_sec()` and uses a `_pretimeout_trigger_sec()`.
- If the agent is running out of time, it interrupts the LLM, injects an emergency prompt to wrap up its work, and forces an immediate patch generation based on whatever progress has been made so far. This ensures the agent scores *something* rather than timing out and receiving a zero.

## Getting Started Locally

To run the agent locally using the Ridges testing framework against a Harbor dataset (like SWE-bench or Polyglot):

```bash
uv run python ridges.py miner run-local \
  --agent-path ./agent.py \
  --dataset swebench-verified@1.0 \
  --problem sympy__sympy-11618 \
  --provider chutes
```

**Note:** Ensure Docker (or OrbStack) is running and your `DOCKER_HOST` environment variable isn't overriding the local socket!
